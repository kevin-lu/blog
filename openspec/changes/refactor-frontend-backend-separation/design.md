## Context

**当前状态**:
- 项目使用 Nuxt.js 全栈框架，前后端耦合在同一代码库
- 启动时经常出现依赖冲突、构建错误
- SSR 渲染在本地开发和生产环境表现不一致
- 数据库使用 SQLite + Drizzle ORM，运行正常

**约束条件**:
- 数据库连接方式保持不变 (SQLite 文件)
- 现有数据需要完整迁移
- 保持现有功能完整性 (博客展示、管理后台、评论等)
- 分步迁移策略，确保业务连续性

**利益相关者**:
- 开发者：需要更清晰的代码结构和更简单的开发流程
- 用户：无感知迁移，功能保持一致
- 运维：简化部署流程，降低维护成本

## Goals / Non-Goals

**Goals:**
- 实现前后端完全分离，独立开发、独立部署
- 后端采用 Flask + SQLAlchemy 提供 RESTful API
- 前端采用 Vue 3 + Vite 构建纯 SPA 应用
- 实现 JWT 认证系统管理后台权限
- 提供 Swagger API 文档方便开发调试
- 添加监控日志和性能指标收集
- 实现 Redis 缓存层提升性能
- 支持文件上传和管理功能

**Non-Goals:**
- 不改变数据库类型 (保持 SQLite)
- 不改变现有业务逻辑和功能
- 不引入 GraphQL 或 tRPC 等其他 API 风格
- 不支持实时通信 (WebSocket)
- 不重构 Sanity CMS 集成 (保持现有)

## Decisions

### 1. 后端框架选择：Flask vs Django vs FastAPI

**决策**: 选择 Flask

**理由**:
- Flask 轻量级、灵活，适合中小型项目
- 学习曲线平缓，易于上手
- 生态系统成熟，扩展丰富
- 可以按需选择组件，避免 Django 的过度封装
- FastAPI 虽然现代但团队熟悉度低

**替代方案**:
- Django + DRF: 功能全面但过于重量级，很多用不到
- FastAPI: 异步特性好但需要团队学习新技术

### 2. 前端架构：纯 Vue 3 vs Nuxt.js

**决策**: 纯 Vue 3 + Vite

**理由**:
- 避免 Nuxt.js 的复杂性和启动问题
- 纯 SPA 架构更简单，调试更容易
- Vite 构建速度快，开发体验好
- 配合 Naive UI 组件库快速开发

**替代方案**:
- Nuxt.js (仅前端模式): 仍可能有 SSR 相关问题
- VitePress: 适合文档型网站，不适合博客管理后台

### 3. API 认证：JWT Token vs Session

**决策**: JWT Token + Refresh Token 机制

**理由**:
- 无状态认证，适合分离架构
- 前端存储 Token，后端验证签名
- 支持跨域请求，无需处理 Session 共享
- Refresh Token 机制提升安全性

**实现细节**:
- Access Token: 15 分钟有效期
- Refresh Token: 7 天有效期，HttpOnly Cookie
- Token 刷新接口：`POST /api/auth/refresh`

### 4. 数据库 ORM: SQLAlchemy vs Tortoise ORM

**决策**: SQLAlchemy 2.0+

**理由**:
- Python ORM 事实标准，文档丰富
- 支持同步和异步两种模式
- 类型提示友好 (配合 SQLAlchemy 2.0)
- 社区活跃，问题容易解决

**实现细节**:
- 使用 SQLAlchemy 2.0 的 `select()` 语法
- 配合 `sqlmodel` 或原生 SQLAlchemy
- 保持现有数据表结构不变

### 5. 部署架构：完全分离 vs 统一托管

**决策**: 前后端完全分离部署

**架构**:
```
┌─────────────────┐         ┌─────────────────┐
│   Vercel /      │         │   云服务器 /     │
│   Netlify       │         │   Docker        │
│   (前端静态)    │◄───────►│   (Flask API)   │
└─────────────────┘   CORS  └────────┬────────┘
                                     │
                                     ▼
                              ┌───────────────┐
                              │   SQLite DB   │
                              │   (本地文件)   │
                              └───────────────┘
```

**理由**:
- 前端利用 CDN 加速，加载更快
- 后端专注 API 服务，资源独立
- 故障隔离，互不影响
- 扩展灵活，可独立扩容

### 6. API 文档：Swagger/OpenAPI

**决策**: Flask-RESTX 或 Flasgger

**理由**:
- 自动生成 OpenAPI 规范
- 内置 Swagger UI 界面
- 支持在线测试 API
- 代码即文档，保持同步

### 7. 缓存策略：Redis

**决策**: Redis 缓存层

**缓存内容**:
- 文章列表 (分页)
- 热门文章统计
- 分类/标签聚合数据
- 用户 Session (可选)

**缓存失效**:
- 文章更新时清除对应缓存
- 定时刷新热点数据
- 手动清除接口 (管理后台)

### 8. 文件存储：本地 + CDN

**决策**: 本地存储 + CDN 加速

**实现**:
- 上传文件存储到本地 `uploads/` 目录
- 使用 Flask-Uploads 管理文件
- 图片自动生成缩略图
- CDN 加速静态资源访问

## Risks / Trade-offs

### 风险 1: 数据迁移丢失

**风险**: 从 Drizzle ORM 迁移到 SQLAlchemy 时可能丢失数据或表结构不一致

**缓解措施**:
- 先导出所有数据为 JSON/CSV 备份
- 编写数据迁移脚本验证数据完整性
- 在测试环境充分验证后再上线
- 保留回滚方案 (保持旧代码可用)

### 风险 2: API 接口不兼容

**风险**: 新 API 接口与前端调用不匹配，导致功能异常

**缓解措施**:
- 先实现 API 接口，用 Swagger 测试验证
- 前端使用 Mock 数据开发，最后联调
- 保持 API 版本控制 (`/api/v1/`)
- 编写 API 集成测试用例

### 风险 3: 认证安全问题

**风险**: JWT Token 泄露或被盗用

**缓解措施**:
- Token 使用 HttpOnly Cookie 存储
- 实现 Token 黑名单机制
- 添加 IP 绑定验证
- 定期轮换 JWT Secret
- 实现登录失败次数限制

### 风险 4: 性能下降

**风险**: 前后端分离后网络请求增加，可能导致性能下降

**缓解措施**:
- 实现 Redis 缓存层减少数据库查询
- 前端实现请求合并和防抖
- API 实现分页和字段过滤
- 使用 Gzip 压缩响应数据
- CDN 加速静态资源

### 风险 5: 开发复杂度增加

**风险**: 需要同时维护前后端两个项目，开发复杂度上升

**缓解措施**:
- 使用 Docker Compose 统一本地开发环境
- 编写详细的开发文档
- 实现自动化测试和 CI/CD
- 分步迁移，逐步适应新架构

### 风险 6: CORS 跨域问题

**风险**: 前后端分离后出现 CORS 跨域错误

**缓解措施**:
- 后端配置 Flask-CORS 允许前端域名
- 生产环境使用 Nginx 反向代理统一域名
- 正确配置 Cookie 的 SameSite 和 Secure 属性
- 实现预检请求缓存
